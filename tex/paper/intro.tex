\section{Introduction}\label{sec:intro}

The \bmethod~\cite{abrial1996bbook} is a formal method for developing software
by \emph{stepwise refinement}: a development starts from an abstract machine
(state, invariants, and operations in first-order logic and set theory) and
proceeds through proved refinement steps to executable code.
%
% TODO. what about the metro line 1? seperate citation?
It has a long industrial record in the verification of safety-critical systems,
most prominently rail transport: the automatic train control of Paris métro
line~14 was developed in B~\cite{behm1999meteor}.
%
\ab~\cite{clearsy_atelierb} is a software package supporting the method,
developed and distributed by CLEARSY.
%
For every machine and refinement step, Atelier B generates
\emph{proof obligations} that establish the preservation of invariants,
well-definedness, absence of arithmetic overflow, etc.
%

\ab's main prover applies rules drawn from a large rule
base~\cite{lecomte2024provingb}, and users routinely extend this base
with bespoke rules capturing the reasoning of their domain; an added
rule must itself be validated.
%
The \emph{Predicate Prover} (\pp) is \ab's automated theorem prover,
and serves both roles: it discharges proof obligations directly, and it
is the standard tool for validating added rules.
%
Despite B's set-theoretic surface, \pp{} is almost entirely a prover for
first-order logic with linear arithmetic: a pre-processing pass reduces
the set theory of an obligation to first-order formulas over membership,
and the set operators that remain reach \pp{} as uninterpreted function
symbols.
%
\pp{} is closed-source: its verdicts are trusted by \ab{} without
independent verification.
%
This is in contrast to setups like Isabelle's
Sledgehammer~\cite{sledgehammer}, where proofs found by external provers
are reconstructed within the proof assistant's kernel rather than taken
on faith.
%
Fortunately, \pp{} is capable of recording a compact \emph{trace} listing the
inference rules it applied during a proof.
%
Another utility expands this trace into a \emph{replay} that annotates
each step with the intermediate goal it was applied to.

%
Our tool \tool, reconstructs these replays in \lp~\cite{hondet:hal-02981561},
a proof assistant based on the \lpcmr~\cite{CousineauDowek}.
%
The pipeline is visualized in \autoref{fig:pipeline}.
%
We encode \pp's inference rules as \lp{} symbols whose types capture their
premises and conclusions, and whose bodies prove them sound with respect to
a portion of \lp{}'s standard library covering classical logic, linear
arithmetic, and a few axioms for basic set theory.
%
We evaluate our tool on benchmark suites containing rule validations
and proof obligations from real-world \ab{} projects, as well as a large
synthetic suite of goals generated by Claude Opus 4.8.
%

\paragraph{Related Work.}
This work is carried out within the ANR project ICSPA (\emph{Interoperable
and Confident Set-based Proof Assistants})\footnote{\url{https://icspa.inria.fr}},
which aims to connect the set-theoretic proof systems of industrial formal
methods (B and TLA$^+$) to the \dedukti{} ecosystem for independent verification
and interoperability.
%
Sibling efforts in the project encode Event-B's mathematical language
and its Rodin-produced proofs in \lp~\cite{grieu2024eventb,
grieu2025jfla}, and translate Event-B developments to
TLA$^+$~\cite{grieu2025eventbtla}.
% TODO(authors): Catherine's AFADL 2026 companion paper (Lille, 2--3
% June 2026) has no HAL record yet — add the citation once the actes
% are deposited (nothing to cite as of 2026-06-12).
The work presented here complements these approaches, since \pp{} is
used in Rodin.

The reduction of proof checking to type checking originates with the
Edinburgh Logical Framework~\cite{harper1993lf} and its dependently-typed
$\lambda\Pi$-calculus.
%
\dedukti{} extends this calculus with rewrite rules,
yielding the \lpcmr~\cite{CousineauDowek, assaf2023dedukti}, of which \lp{} is
the interactive front end~\cite{hondet:hal-02981561}.
% TODO. missing Melanie Taprogge's work on LEO-III.
% probably others. search web for ATP encodings in dedukti/lambdapi.
Replaying automated provers inside such checkers is an active line of research:
SMT proofs have been reconstructed in \lp~\cite{coltellacci2026acta}
(including proofs of TLAPS obligations, TLA$^+$'s proof system~\cite{coltellacci2023abz})
and in Lean~\cite{mohamed25leansmt}.
%
The same idea underlies \emph{hammer} tools, which discharge a goal with
external automated provers and rebuild the returned proof in the assistant's
kernel, as Isabelle's Sledgehammer~\cite{sledgehammer} does for its SMT and
ATP backends.

Confidence in \ab's proof machinery has been pursued from several
directions.
%
Cirstea and Kirchner formalised \pp's inference rules in ELAN~\cite{cirstea1998elan}.
Their encoding re-implements the prover's first-order core, but covers
neither arithmetic nor any soundness argument for the rules themselves.
%
Jaeger and Dubois~\cite{jaeger2007trustb} give a deep embedding of B's logic
in Rocq (BiCoq), and the BCARe framework of Jacquel et~al.~\cite{jacquel2011bcare, jacquel2015sosym}
works over a similar embedding to verify the rules added by users
to \ab's rule base.

The BWare project~\cite{bware2014} addressed the same concern differently,
discharging B proof obligations with external provers (notably Zenon Modulo,
whose proofs are checked by \dedukti~\cite{bury2015bset}) and thereby replacing
\pp{} rather than formally verifying its output.
%
In the same spirit, Trélat's BEer~\cite{trelat2025smt} gives a provably
sound encoding of B proof obligations into SMT-LIB, as does
Defourné~\cite{defourne2025scp} for TLA$^+$ obligations, while Bergeron
and Trélat's BARReL~\cite{bergeron2026barrel} reimplements \ab's proof
backend in Lean.
%
Our work targets the actual deductions performed by \pp{}: each proof replay
is encoded and checked against rule lemmas that are themselves proved sound.
