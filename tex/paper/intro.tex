\section{Introduction}\label{sec:intro}

The \bmethod~\cite{abrial1996bbook} is a formal method for developing
software by stepwise refinement: a development starts from an abstract
machine — state, invariants, and operations expressed in first-order
logic and set theory — and proceeds through proved refinement steps down
to executable code.
%
It has a long industrial record in safety-critical systems, most
prominently in rail: the automatic train control of Paris métro
line~14 was developed in B~\cite{behm1999meteor}.
%
\ab~\cite{clearsy_atelierb} is the toolset supporting the method,
developed and distributed by CLEARSY.
%
For every machine and refinement step it generates \emph{proof
obligations} — sequents whose validity establishes that an invariant is
preserved or a refinement is correct — and discharges them with
automated provers, leaving the remainder to interactive proof.

\ab's main prover applies rules drawn from a large rule
base~\cite{lecomte2024provingb}, and users routinely extend this base
with bespoke rules capturing the reasoning of their domain; an added
rule must itself be validated.
%
The \emph{Predicate Prover} (\pp) is \ab's automated theorem prover for
first-order logic, and serves both roles: it discharges proof
obligations directly, and it is the standard instrument for validating
added rules.
%
\pp{} is closed-source: its verdicts are trusted by \ab{} without
independent verification.
%
This is in contrast to setups like Isabelle's
Sledgehammer~\cite{sledgehammer}, where proofs found by external provers
are reconstructed within the proof assistant's kernel rather than taken
on faith.
%
We address this gap with \tool, a tool that reconstructs \pp{} proofs in
\lp~\cite{hondet:hal-02981561}, a proof assistant based on the
\lpcmr~\cite{CousineauDowek}.
%
We encode each \pp{} inference rule as a \lp{} symbol whose type captures
the rule's premises and conclusion, and whose body is a proof that
demonstrates the correctness of the rule with respect to classical logic.

%
\pp{} can be instrumented to record a compact \emph{trace} listing the
inference rules it applied during a proof; another utility expands this
trace into a \emph{replay} that annotates each step with the rule
applied and the intermediate goal it was applied to.
%
Given a replay, \tool{} parses it, builds a proof tree, and emits a \lp{}
tactic script that composes the encoded rules following the tree, yielding
a complete, independently checkable proof (\autoref{fig:pipeline}).
%
On \ab's own rule-validation corpus and on a large synthetic suite,
every proof the upstream toolchain can replay — \allReplayed{} in total —
is reconstructed and type-checked, and the reconstructed proofs use no
unproved assumption beyond a small, fixed axiomatic base
(\autoref{sec:trust}).
%
This work is carried out within the ANR project ICSPA (\emph{Interoperable
and Confident Set-based Proof Assistants}),\footnote{\url{https://icspa.inria.fr}}
which connects the set-based proof systems of industrial formal methods
— B and TLA$^+$ — to the \dedukti{} ecosystem for independent
certification and interoperability.

\subsubsection*{Related Work}

The reduction of proof checking to type checking originates with the
Edinburgh Logical Framework~\cite{harper1993lf}; \dedukti{} extends the
$\lambda\Pi$-calculus with rewriting~\cite{CousineauDowek,
assaf2023dedukti}, and \lp{} is its interactive front
end~\cite{hondet:hal-02981561}.
%
Replaying automated provers inside such checkers is an active line of
work: SMT proofs have been reconstructed in
\lp~\cite{coltellacciReconstructionSMTProofs2024a} and in
Lean~\cite{mohamed25leansmt}, and Sledgehammer rebuilds external
provers' proofs through Isabelle's kernel~\cite{sledgehammer}.

Confidence in \ab's proof machinery has been pursued from several
directions.
%
Cirstea and Kirchner formalised \pp's inference rules as rewriting
strategies in ELAN~\cite{cirstea1998elan}; their encoding re-implements
the prover's first-order core, but covers neither arithmetic nor any
soundness argument for the rules themselves.
%
The BCARe framework of Jacquel et~al.~\cite{jacquel2011bcare} addresses
the rules \emph{added} by users to \ab's rule base, verifying them in
Coq over a deep embedding of B's logic.
%
Our work is complementary to both: we certify the prover's own
inferences, proof by proof.
%
The BWare project~\cite{bware2014} took a different route to the same
trust concern: it discharged B proof obligations with external certifying
provers — notably Zenon Modulo, whose proofs are checked by
\dedukti~\cite{bury2015bset} — thereby \emph{replacing} \pp{} rather
than certifying it.
%
In the same spirit, Trélat~\cite{trelat2025smt} gives a provably sound
encoding of B proof obligations into SMT-LIB, so that SMT solvers can be
applied to them safely.
%
Closest to our setting, Stolze et~al.~\cite{stolze2024dedukti} also
target \dedukti, working towards exporting and sharing \ab{} proofs;
% TODO(authors): position stolze2024dedukti precisely (which prover's
% proofs, what stage the work is at).
\tool{} differs in certifying \pp's own rule applications against rule
lemmas that are themselves proved sound.
