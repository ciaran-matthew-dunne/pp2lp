The core of
\tool{}\footnote{\url{https://github.com/ciaran-matthew-dunne/pp2lp}}
is an OCaml program of about 4600 lines behind a Python command-line
driver; the rule library of \autoref{sec:encoding} is another 4300
lines of \lp.
%
A single module holds rule metadata: arity, argument kinds,
and an emission strategy per rule.
%
The pipeline (\autoref{fig:pipeline}) has three stages.

A lexer and a Menhir-generated parser read the \texttt{.replay} file
into a flat list of (rule, argument, goal) records.
%
This list is then transformed into a proof tree by consuming the prefix stream
and using each rule's declared arity to attach the right number of child subproofs.
%
The two irregularities noted in \autoref{sec:pp-syntax} are handled
at this stage: normalisation markers are filtered out, and the
result derivation that precedes an instance of $\rn{ALL7}/\rn{XST8}$
is reassembled into that rule's first argument.

%
Each replay becomes a \lp{} \texttt{symbol} whose statement is the goal
and whose body is a tactic script.
%
Each rule application in the \pp{} proof tree generally corresponds to an
application of the corresponding \lp{} rule (e.g., $\refine\;\texttt{AND2\,\_}$),
although some \pp{} rule applications require bespoke tactic applications.
%
Rules that recall in-scope hypothesis receive their $\Prf$-argument
by hypothesis lookup at the OCaml level; rules that introduce an antecedent
or a binder push an \texttt{assume}; result derivations are emitted as
explicit $\Res$-terms and passed as an argument; and arithmetic solver
side-conditions get proof-terms generated at translation time.
(\autoref{sec:enc-rules}).
%
The resulting symbol declaration is written to a \lpf{} file which
may then be type-checked via \texttt{lambdapi check}.
%
For debugging purposes, our tool records a table mapping every line of the
\lp{} file to the replay step and \pp{} goal that produced it.
%
When a check fails, the report shows the offending \lp{} line, the replay
step it came from, and the rule lemma's signature side by side.

\subsection{Benchmarks}\label{sec:bench}

We exercise \tool{} on three suites (\autoref{tab:results}).
%
\texttt{og} is a fixed set of \ogReplayed{} replays that served as an
initial development corpus.
%
\texttt{prv} are goals that validate custom proof-rules from real
\ab{} developments.
%
\texttt{claude} is a synthetic suite of \claudeGoals{} goals, authored by
an LLM-based coding agent (Claude Opus 4.8).

\subsection{Conclusion and Future Work}\label{sec:conclusion}

We have presented \tool, which checks \ab's Predicate Prover by
reconstructing its proofs in \lp.
%
Unlike approaches that postulate each rule as an axiom, every \pp{} rule
is encoded as a symbol whose body \emph{proves} the rule sound against a
small axiomatic base, and the reconstructed corpus (including \ab's own
rule-validation suite, arithmetic included) type-checks without a
single appeal to an oracle.
%
The remainder of this section discusses what limits coverage today and
where the work goes next.

\paragraph{Upstream replay losses.}
The gap between goals and replays in \autoref{tab:results} is upstream
of \tool: for some provable goals the \textsc{replay} utility omits the
final continuation of an \rn{ALL7} or \rn{XST8} step, and for others it
aborts mid-file.
%
Both defects are detected at generation time and the affected goals
dropped and reported.
%
They are not random: set-extensionality and equality-prover-heavy goals
are disproportionately affected (no complete \rn{EGALITE} replay can be
produced from our goal suites, so that emission path is validated
against a hand-completed replay), which means the suites under-test
exactly the rules the equality prover would exercise.

\paragraph{Spec mismatches.}
Encoding every rule against the specification turns the encoding into an
executable test of that document, and it does not always pass: \rn{NRM20}
records its pinned equality at a different conjunct position than
specified, and \rn{OPR1}'s result derivation does not follow the schema
laid out for it.
%
\tool{} implements the observed behaviour; we believe such discrepancies
are themselves a useful by-product for the maintainers of \pp.

\paragraph{Engine partiality.}
A few replay shapes are recognised but rejected: substitution rules
pinning a binder beyond the third slot of a compound binder, and the
result forms of two arithmetic rules that no replay in our suites
exercises.
%
All of these are rejected at emission with an error; none silently degrade.

\paragraph{Raw proof obligations.}
We also probed \pp{} on raw industrial obligations from CLEARSY's
public proof-obligation dataset~\cite{deharbe2022pog}, translating its
POG format into \pp's rule-validation input.
%
A sampled obligation carries a median of roughly \aperoMedianHyps{}
hypothesis predicates; \pp{} nonetheless proved every convertible
obligation with at most fifty hypotheses, most sampled goals even with
all hypotheses removed, and some thousand-hypothesis obligations given
tens of seconds.
%
The bottleneck for certification is again the toolchain around \pp: the
replay utility lost \aperoSampleLost{} of the \aperoSampleTraces{}
traces \pp{} produced on our sample, and the replays that remain use
surface forms no synthetic goal had reached (strict comparisons, tuple
memberships, relational inverse, sequents inside annotations);
\aperoSampleVerified{} of them verify end-to-end today, including the
first machine-produced replays exercising the equality store.

\paragraph{Use of AI.}
The \texttt{claude} suite was authored by an LLM agent that also helped
develop the tool, so there is a risk of the suite drifting toward what
the pipeline already handles.
%
We mitigate this in two ways: the suite is goal-level only (the agent
cannot influence which rules \pp{} applies), and the real-world
\texttt{og} and \texttt{prv} suites gate every change.

\paragraph{Future work.}
The most direct future work is upstream: repairing the \textsc{replay}
utility's truncations would recover the dropped goals, in particular the
equality-prover fragment.
%
On the encoding side, the binder-regrouping rules are currently
consumed at the syntax level rather than emitted
(\autoref{sec:enc-formulas}); deriving the regrouping isomorphisms over
tuples would let \tool{} emit them like any other rule and drop its one
syntax-level normalisation.
%
Finally, this work folds \pp{} into the \dedukti{} interoperability
ecosystem: a natural next step within the ICSPA project is exporting
checked \ab{} proofs to set-theory-based assistants such as TLA$^+$'s
TLAPS, Isabelle/ZF, and Megalodon.
